## Redis基本数据类型和结构

- **String：**String 类型的底层的数据结构实现主要是 **SDS（简单动态字符串）**。 SDS 和我们认识的 C 字符串不太一样，之所以没有使用 C 语言的字符串表示，因为 SDS 相比于 C 的原生字符串：
  - SDS 不仅可以保存文本数据，还可以保存二进制数据。因为 SDS 使用 len 属性的值而不是空字符来判断字符串是否结束，并且 SDS 的所有 API 都会以处理二进制的方式来处理 SDS 存放在 `buf[]` 数组里的数据。所以 SDS 不光能存放文本数据，而且能保存图片、音频、视频、压缩文件这样的二进制数据。
  - SDS 获取字符串长度的时间复杂度是 O(1)。因为 C 语言的字符串并不记录自身长度，所以获取长度的复杂度为 O(n)；而 SDS 结构里用 len 属性记录了字符串长度，所以复杂度为 O(1)。
  - Redis 的 SDS API 是安全的，拼接字符串不会造成缓冲区溢出。因为 SDS 在拼接字符串之前会检查 SDS 空间是否满足要求，如果空间不够会自动扩容，所以不会导致缓冲区溢出的问题。
- **List：**在 Redis 3.2 版本之后，List 数据类型底层数据结构就只由 quicklist 实现了，替代了双向链表和压缩列表。
- **Hash：**在 Redis 7.0 中，压缩列表数据结构已经废弃了，交由 listpack 数据结构来实现了。
- **Set：**如果集合中的元素都是整数且元素个数小于 512 （默认值，set-maxintset-entries配置）个，Redis 会使用整数集合作为 Set 类型的底层数据结构；如果集合中的元素不满足上面条件，则 Redis 使用哈希表作为 Set 类型的底层数据结构。
- **Zset：**在 Redis 7.0 中，压缩列表数据结构已经废弃了，交由 listpack 数据结构来实现了。

## ※※Redis 主从同步是怎样的过程?

Redis的主从同步过程可以分为以下几个阶段:

1.连接建立:

。从节点启动时，会通过配置的主节点地址和端口号，向主节点发送SYNC命令，请求建立复制关系。2.全量同步(初次同步):

。主节点接收到SYNC命令后，会执行BGSAVE命令，在后台生成一个RDB文件(快照文件)，同时继续接收和处理客户端请求，并将这些写命令记录在缓冲区中。

当RDB文件生成完成后，主节点会将文件发送给从节点。

。从节点接收到RDB文件后，会清空自己的数据集，然后加载RDB文件，完成数据集的初始化。

主节点会将缓冲区的写命令发送给从节点，从节点依次执行这些命令，确保数据集与主节点一致。3.部分同步(增量同步):

。全量同步完成后，主从节点进入命令传播阶段。主节点会将后续接收到的写命令实时发送给从节点，从节点依次执行这些命令，保持数据集的同步。

如果从节点因网络原因断开连接，重新连接后会尝试进行部分同步。主节点会根据从节点发送的复制偏移量(replication offset)，判断从节点的数据丢失情况，然后发送丢失的部分数据(通过发送缺失的命令或者部分RDB文件)给从节点，完成数据的恢复和同步。

4.心跳检测:

。主从节点之间会定期发送心跳包，检测连接状态和同步进度。主节点会记录每个从节点的复制偏移量和最后同步时间，从节点会汇报自己的复制偏移量和状态信息。

总结来说，Redis的主从同步过程包括全量同步和部分同步两个阶段，通过RDB文件和命令传播实现数据的复制和同步，同时通过心跳检测机制保持连接的稳定性和数据的实时性。

## ※Redis的aof文件过大怎么处理，怎么解决，开启aof的方式有几种?

当Redis的AOF(Append Only File)文件过大时，可以采取以下几种处理方式:

处理方式

1.自动重写(AOF重写):

。Redis提供了自动重写机制，当AOF文件大小超过一定阔值时，会自动触发AOF重写。重写过程中，Redis会生

成一个新的AOF文件，只包含恢复数据集所需的最少命令，从而减小文件大小。可以通过配置auto-aof-rewrite-percentage和auto-aof-rewrite-min-size参数来控制AOF重写的触发条件。2.手动触发重写:

。可以通过执行BGREWRITEAOF命令手动触发AOF重写，无论当前文件大小是否达到阔值。3.调蝥AOF策路:。可以通过调整appendfsyc参数来控制AOF文件的同步频率，从而影响文件大小和性能。appendfsync always同步频率最高，文件大小最小，但性能影响最大;appendfsynceverysec是默认配置，性能和安全性较为平衡:appendfsyncno则性能最好，但数据丢失风险最大。4.定期清理:

。如果AOF文件过大且不需要保留历史数据，可以考虑手动制除AOF文件，但需要谨慎操作，确保数据安全

开启AOF的方式

1.配置文件:

。在Redis配置文件中，将appendonly参数设置为yes，开启AOF持久化。

。可以设置appendfilename参数指定AOF文件名，appendfsync参数控制同步频率等。2.命令行参数:

。启动Redis时，可以通过--appendonlyyes参数开启AOF持久化。3.动态配置:

。在Redis运行时，可以通过CONFIGSET命令动态修改appendonly参数开启或关闭AOF持久化。

总结来说，处理Redis AOF文件过大的问题主要依靠自动和手动重写机制，同时可以通过调整AOF策略和定期清理来优化文件大小。开启AOF的方式主要有配置文件、命令行参数和动态配置三种。

## Redis限流器可以怎么设计?

Redis限流器的设计可以利用Redis的高速读写能力和原子作特性，以下是几种常见的设计思路:

基于Redis的计数器限流

1.园定窗口计数器:。设定一个周定时间窗口(如1秒)，使用Redls的INCR命令统计该时间窗口内的请求次数如果请求次数超过设定的阔值，则拒绝后续请求，直到时间窗口结束并重置计数器。。可以使用Redis的EXPIRE命令设置时间窗口的过期时间，自动清理过期的计数器。2.滑动窗口计数器:

将固定时间窗口分成多个小窗口(如10个100毫秒的小窗口)，每个小窗口独立计数。

通过维护一个小窗口的列表，并计算列表中所有小窗口的请求总数，实现滑动口计数。

可以使用Redls的有序集合(Sorted Set)来存储每个小窗口的计数和时问戳，通过ZREMRANGEBYSCORE命

令副除过期的小窗口。

基于Redis的令牌桶算法限流

1.令牌桶:

。维护一个因定容量的令牌桶，以固定速率向桶中添加令牌。

每个请求到达时，从令牌桶中获取一个令牌，如果获取成功则允许请求通过，否则拒绝请求。

可以使用Redis的SETNX和DECR命令实现令牌的添加和获取，以及使用Redis的INCRBY命令以图定速率增加令牌效量。

基于Redis的漏桶算法限流

1.漏桶:

设定一个周定容量的漏桶，以周定速率从杨中漏出请求(即处理请求)。

每个请求到达时。首先放入漏桶，如果漏橘已满则拒绝请求。

可以使用Redis的LIST结构实现漏桶，使用LPUSH命令添加请求，使用RPOP命令漏出请求，并使用Redis的定

时任务或Lua脚本控制漏出速率。

其他优化

1.分布式阴流:在分布式系统中，可以使用Redis的分布式锁或分布式计数实现全局限流，确保限流策略的一致性。

2.动态调整:根据实际流里情况，动志调整限流词值和令牌生成速率，提高限流器的灵活性和适应性。

3.日志记录:记录被拒绝的请求和相关信息，用于篮控和分析限流效果，以及排查潜在问题。

总结来说，Redis限流器的设计主要基于计数器、令牌桶和漏桶算法，利用Redis的高速读写和原子操作特性实现高效的限流功能。同时，可以根据实际需求进行优化和扩展，如分布式限流和动态调整等。

### redis的缓存失效如何进行回源查询

1. 在应用程序中实现回源逻辑
   1. **查询缓存**：应用程序首先向Redis发送查询请求，如果Redis中未命中（即缓存失效），则触发回源查询。
   2. **回源查询HBase**：使用HBase的Java API或相关的客户端库，构建查询语句来获取所需的数据。例如，通过指定表名、行键等信息来查询HBase中的具体记录。
   3. **更新缓存**：将从HBase查询到的数据更新到Redis缓存中，以便后续查询可以直接从缓存中获取数据，提高查询效率。同时，设置合适的缓存过期时间，避免缓存数据长时间不更新而导致数据不一致。
2. 使用缓存框架的回源策略

- 一些缓存框架如Spring Cache等提供了缓存失效后的回源查询策略配置。可以在配置文件中指定当缓存失效时，自动调用指定的方法来查询数据源（如HBase），并将结果缓存起来。这样，在应用程序中只需要简单地使用缓存注解，框架会自动处理缓存失效后的回源查询逻辑，简化了代码实现。

1. 在实现回源查询时，还需要考虑一些问题，如HBase查询的性能优化、缓存穿透（大量不存在的键请求导致每次都回源查询）的防范等，以确保系统的稳定性和性能。
